ST17 SCOPE
------------------------------------------------------------------
1. Public profile page apps/web/app/users/[username]/page.tsx — client component at
   /users/<username>; fetches GET /api/users/:username; renders ONLY five fields
   (username, displayName, avatar, bio, joinDate). A pure, exported profileProjection()
   enforces the five-field allowlist at runtime (drops unknown keys; returns null if a
   required field is missing). 404 -> "Profile not found". All text via React text nodes
   (no dangerouslySetInnerHTML).
2. Avatar upload/replace/remove in apps/web/app/profile/page.tsx — ImageUpload
   resourceType="avatar" -> POST /api/media/upload?resourceType=avatar; on success
   PUT /api/users/me/avatar (ST15) to bind mediaId; Remove -> DELETE /api/users/me/avatar.
   Client control is UX only; ST15 API is the real enforcement boundary.
3. Shared UserAvatar component apps/web/components/user-avatar.tsx — used on public profile,
   ST16 forum bylines, and mention-autocomplete. Pure resolveAvatarSrc()/deriveInitials().
   Renders <img> only when avatarSrc is non-null AND not errored; otherwise an uppercase
   initials span. onError -> imgError -> initials (no broken image, no raw URL in DOM).
4. auth-client.ts setAvatar(mediaId)/removeAvatar() -> PUT/DELETE /api/users/me/avatar.
5. mention-autocomplete.tsx uses UserAvatar for result rendering; inserts plain-text @username.

Confirmed PASS criteria:
- No PII leak beyond the five allowlisted profile fields. Client profileProjection() drops
  email/globalRole/status/id/passwordHash and returns null on missing required field; this is
  defense-in-depth over the ST14 server-side allowlist (users.service.ts findPublicProfile
  returns exactly five fields).
- Avatar serve path is the gated /api/media/<id> URL (built server-side from avatarMediaId),
  never a raw storage URL. resolveAvatarSrc returns null (initials fallback) when src is
  falsy OR hasError; <img> is rendered only after the null guard; onError degrades to initials.
- Every /users/<username> link is encodeURIComponent-encoded (public-profile fetch, suggest
  fetch, ST16 byline post/topic author links). ST17 introduced no un-encoded link.
- Avatar bind/remove uses the ST15 API as the real boundary: users.service.ts setAvatar queries
  id=mediaId AND resourceType='avatar' AND ownerUserId=callerId and throws a uniform
  ForbiddenException (oracle parity) otherwise — a client-side bypass passing another user's
  mediaId is blocked SERVER-SIDE, not by the client.
- No dangerouslySetInnerHTML / innerHTML anywhere in the ST17 surface.
