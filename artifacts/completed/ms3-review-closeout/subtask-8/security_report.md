## Security assessment summary

The fix is correct, layered, and fully closes the oracle this subtask targeted. All three security acceptance criteria are met.

- **Oracle closed at the controller layer.** `resolvePostId` (`blog.controller.ts:354-363`) now applies the identical published-only visibility predicate on both branches: `findPublishedBySlug` then `findPublishedById`. Both service methods enforce `status: "published", publishedAt: LessThanOrEqual(now)` (`blog.service.ts:109-115, 124-130`). Nonexistent, draft, unpublished, and future-scheduled posts — addressed by slug or UUID — all reach the single `throw new NotFoundException("Blog post not found.")` at `blog.controller.ts:362`. One throw site means status code, message, and envelope are identical by construction.
- **Oracle closed again at the service layer (defense-in-depth).** `createComment` (`blog.service.ts:377-387`) re-fetches the post (any status) and re-evaluates the publication predicate at write time, throwing `NotFoundException("Blog post not found.")` — the same class and message string as the controller's throw and as the genuinely-nonexistent case (`blog.service.ts:379`). This makes the write-time check authoritative: even the TOCTOU window between `resolvePostId` and `createComment` cannot leak (a post that turns non-public mid-request yields the identical 404), and a future regression in either layer alone leaves the invariant intact.
- **Envelope identity confirmed.** `JsonExceptionFilter` builds the error payload purely from exception class + message (`json-exception.filter.ts:58-102`); both 404 paths produce byte-identical `error` objects (`code: "NOT_FOUND"`, `message: "Blog post not found."`, `statusCode: 404`) with only the per-request correlation id/timestamp differing — which differ on every request regardless.
- **Guard ordering prevents a secondary oracle.** The publication check (`blog.service.ts:382-387`) executes before the `commentsLocked` check (`:390-392`), so a locked draft/unpublished/future-scheduled post returns 404, never the lock 403. The lock 403 is reachable only for posts that already passed the public-visibility predicate — a legitimately visible post — so it is not an oracle, exactly as decision D8 specifies.
- **In-memory predicate matches the SQL predicate, fail-closed.** The defense-in-depth check `post.status !== "published" || !post.publishedAt || post.publishedAt > now` is logically equivalent to the SQL `status = "published" AND publishedAt <= now` (boundary equality `publishedAt == now` is allowed in both; `NULL`/falsy `publishedAt` is excluded in both). No state slips between layers.
- **No practical timing channel.** Both the nonexistent-UUID and non-public-UUID cases execute the same two indexed lookups (slug miss, then id miss-or-predicate-excluded); the residual variance ("PK row exists but predicate-excluded" vs "no PK row") is a single indexed-lookup difference, not realistically measurable — same conclusion as the WA1 review of the GET route.
- **All other comment routes behaviorally unchanged.** The diff touches no other handler: `listComments` (public GET, already 404-aligned by the prior subtask-4), `moderationListComments`, `moderateCommentStatus`, `deleteComment`, and `lockComments`/`unlockComments` are byte-identical to the merge-base. The remaining `findById` call sites in the controller are admin-gated (`adminGetById`, line 99) or moderation-gated (`moderationListComments`, line 299) — appropriate any-status surfaces.
- **Web client unaffected.** `blog-client.ts` `createComment` (lines 254-272) is status-agnostic (`!response.ok` → surface server message); the 403→404 contract change cannot break it, addressing plan risk item 8.
- **Strictly narrowing change.** The change removes information (the 403/404 distinction) from an authenticated surface and adds no new query construction, no new input handling, and no new privilege paths.

### NOTE
1. **Verifier deferred-register citation is inaccurate; the TOCTOU deferral is recorded only in verifier artifacts.** The pass-2 verifier report states the `resolvePostId`→`createComment` TOCTOU window is "Deferred per docs/deferred-tasks.md", but `docs/deferred-tasks.md` contains no such entry (its only TOCTOU entry, line 16, is the unrelated `deriveUniqueSlug` one). Security impact is nil — the defense-in-depth write-time check is authoritative, so the race cannot produce an oracle or admit a comment onto a non-public post; per repo policy the register is only editable in a planning cycle anyway. **Planner follow-up:** at the next planning cycle either add the register entry or, preferably, record that no deferral is needed because the service-layer re-check closes the window (this review's assessment: the TOCTOU is moot for the oracle; the only residual race is a comment landing milliseconds before an unpublish/lock, which is equivalent to one landing milliseconds before — a moderation nicety, not a security gap).
2. **The controller-layer wiring is pinned only by simulated tests; the tested authoritative layer is the service guard.** `blog.controller.test.ts:44-199` replicates the `resolvePostId` slug-then-id logic with spies rather than invoking the private method, so a revert of line 360 back to `findById` would not fail any runtime test. Acceptable because (a) the service-level oracle-parity suite pins the authoritative defense-in-depth layer — a non-public post would still produce the identical 404 — and (b) reverting the service guard alone fails the oracle-parity message-identity test. The invariant always has at least one tested enforcement layer. Carry-forward of the verifier NOTE; no action required.
3. **Cross-route 404 message variance (informational, not an oracle).** The POST route's 404 message is "Blog post not found." while the GET `listComments` 404 message is "Blog post not found or not published." Within each route the message is constant across nonexistent/non-public cases, so no information leaks; the variance is cosmetic. The POST route's `@ApiNotFoundResponse` description ("Post not found or not published.") is a description of when the 404 occurs, not the envelope string — accurate as documentation.
4. **Pre-existing authenticated `parentId`/`imageId` distinguisher messages on public posts (out of scope; not worsened).** After the post passes the visibility gate, `createComment` returns distinct 400 messages for "Parent comment not found." vs "Parent comment does not belong to this post." (`blog.service.ts:398-403`) and for nonexistent vs wrong-scope media (`:413-419`). An authenticated member can thereby confirm that an arbitrary comment UUID or media UUID exists somewhere — including rows belonging to since-unpublished posts — which is a much weaker, second-order existence signal. Exploitability is very low (random UUIDs, 400-level, no content disclosure). Candidate for the same data-minimization/deferred bucket as register line 17 at a future planning cycle.
5. **Known items not re-reported:** slug/UUID namespace shadowing in the dual lookup (fail-closed; WA1 NOTE 3), public comment payload data-minimization (register line 17), and the 600-char source-contract window headroom (pass-2 verifier NOTE) — all unchanged by this subtask.
