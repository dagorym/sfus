## Implementation Context

The defect was: `parseInt("abc", 10)` and `parseInt("", 10)` both return `NaN`. The old code did `query.limit ?? RECENT_TOPICS_DEFAULT_LIMIT` — nullish coalescing does NOT catch `NaN` (only catches `null`/`undefined`). So `Math.min(20, Math.max(1, NaN))` returned `NaN`, and `queryBuilder.take(NaN)` threw a `TypeORMError` causing HTTP 500.

The fix in `forums.service.ts`:

```typescript
const rawLimit = query.limit;
const safeLimit = Number.isFinite(rawLimit) ? rawLimit! : ForumsService.RECENT_TOPICS_DEFAULT_LIMIT;
const limit = Math.min(
  ForumsService.RECENT_TOPICS_MAX_LIMIT,
  Math.max(1, safeLimit)
);
```

Note: The controller already passes `parseInt(limit, 10)` to the service when `?limit` is present (so `?limit=abc` results in service receiving `{ limit: NaN }`). The fix is entirely in the service layer. Controller tests mock the service, so they test that the controller does not crash before reaching the service, and that it correctly delegates the NaN value.
