Findings

BLOCKING
- apps/api/src/common/throttle/link-limit.ts:24,36 - BLOCKING — Catastrophic backtracking (ReDoS) in MARKDOWN_LINK_RE = /\]\([^)]*\)/g via countLinks()/exceedsLinkLimit(). Measured on this branch: a Markdown body consisting of the repeated 2-char sequence "](" with no closing paren scales quadratically (O(n^2)): 8 KB ~21 ms, 16 KB ~72 ms, 32 KB ~287 ms, 64 KB ~1,169 ms; a 1 MB body blocked a single node call for ~325 SECONDS in the probe. Each "](" start position matches "](" then greedily consumes the rest with [^)]* and backtracks char-by-char searching for a ')' that never appears, repeated at every position.
  This is the exact FAIL-grade defect the task names ('a ReDoS in the link limiter'). countLinks/exceedsLinkLimit is the reusable per-post link-count control that ST9 wires directly onto fully attacker-controlled Markdown bodies (forum topic/post create, POST /api/blog/:postId/comments). The API is single-instance / single-threaded per the plan topology; NestJS's default JSON body limit (~100 KB) lets an attacker submit a ~100 KB '](' body that blocks the event loop for multiple seconds per request, and a few concurrent such requests deny service to ALL users — turning the anti-abuse control into an attack amplifier. Latent in ST8 (unwired) but exposed the moment ST9 attaches it. Remediation: make the link counter linear — e.g. count '](' occurrences directly (indexOf/split or a non-backtracking token scan) or bound/validate the destination with a possessive/atomic or anchored pattern, plus cap the inspected body length before regex evaluation.

WARNING
- apps/api/src/common/throttle/link-limit.ts:24,28,36 - CONCERN — Link counter only recognizes Markdown ']( ... )' destinations and bare http(s):// URLs. Scheme-less domains ('evil.com'), 'www.evil.com', and non-http schemes ('ftp://', 'mailto:', 'tel:') all count as 0 (verified by probe). A spammer can paste many scheme-less domains and bypass the link cap entirely.
  The link limit is a stated anti-spam baseline (design §13). If the Markdown renderer auto-links bare domains/www hosts (common Markdown behavior) the rendered output is live links the limiter never counted, weakening the control. Not blocking on its own — captcha and bad-word list are explicitly deferred to M11 and this is defense-in-depth — but the gap should be recorded and revisited when ST9 wires enforcement, ideally aligning what the counter sees with what the renderer actually linkifies.
- apps/api/src/common/throttle/throttle.guard.ts:66,72-78,86 - CONCERN — The new-account tier is never enforced through the wired guard. ThrottleGuard always passes userCreatedAt: null (it is declared `const userCreatedAt: Date | null = null` and never reassigned), and AuthService.resolveSession() does not return createdAt. ThrottleService.checkRequest() therefore always computes isNewAccount=false for guard-driven requests, so every authenticated user gets the laxer maxHits, never newAccountMaxHits.
  ST8 AC and design §13 call for a stricter young-account tier. The service logic is correct WHEN given createdAt (unit-tested), but the only wired path (the guard) cannot supply it, so the control is inert as wired. Compounding this, docs/development/api-conventions.md states 'The userCreatedAt timestamp is supplied by the guard from the session payload' — documentation overstates an active control that the code does not deliver (P1 docs/code drift). Direction of failure is fail-OPEN for the tier (laxer limit), not a bucket-share bypass, and it is unwired in ST8, so this is a CONCERN to forward, not a blocker — but ST9 MUST supply userCreatedAt (e.g. a lightweight user lookup) for the tier to function, and the doc should be corrected to match reality.
- apps/api/src/common/throttle/throttle.guard.ts:53-91 - CONCERN — No throttle.guard.test.ts exists. The guard holds the most security-sensitive behavior and none of it is unit-tested: identity resolution from the request, the try/catch around resolveSession (invalid/expired session -> IP fallback, not a 5xx leak), the always-null userCreatedAt, and — critically for R2 — that a store error propagates (fail-CLOSED) rather than being swallowed (fail-open). The InMemoryThrottleStore sweep/eviction path (the memory-exhaustion mitigation) is also untested.
  R2 is explicitly about fail-closed behavior and identity correctness; those guarantees currently rest on code reading, not on regression tests. A future refactor that wraps checkRequest in a swallowing try/catch (a plausible 'don't break requests' change) would silently convert the control to fail-open with no failing test. Add guard tests asserting: store.hit throwing => canActivate rejects/throws (fail-closed); resolveSession throwing => IP identity used and request proceeds; and a store sweep test asserting expired entries are pruned.
- apps/api/.env / apps/api/.env.example / .env / .env.example - CONCERN — The five required THROTTLE_* env vars were added to environment.ts (fail-closed: missing => loadEnvironment throws) and documented in launch.md, but they are NOT present in the git-tracked .env / .env.example / apps/api/.env files. Because config validation fails closed, the API will crash on startup in any environment using the current env files until the vars are added.
  This matches the known 'deploy env drift' failure mode (a crashed API from a missing env var presents as can't-login/can't-register, not an obvious config error). The fail-closed config behavior itself is CORRECT and desirable; the gap is that the env templates were not updated alongside it (ST8's allowed-files list did not include the .env templates). Forward to the coordinator/ST9/deploy owner: add THROTTLE_WINDOW_MS, THROTTLE_MAX_HITS, THROTTLE_NEW_ACCOUNT_MAX_HITS, THROTTLE_NEW_ACCOUNT_WINDOW_MS, THROTTLE_MAX_LINKS_PER_POST to the dev .env and to the production environment before the module's consumers (ST9) ship, or the app will not boot.

NOTE
- apps/api/src/common/throttle/throttle.service.ts:69 - INFO — Identity = userId ?? request.ip ?? "unknown". The literal "unknown" bucket is only reachable when both userId and request.ip are absent; under the production trust proxy=1 topology request.ip is always populated, so the shared 'unknown' bucket is not a practical evasion path. Guest identity rotation is bounded by the proxy topology exactly as intended (no direct XFF parsing).
  Confirms concern #1 resolves PASS: no direct X-Forwarded-For parsing (verified in service + guard + the 'does NOT read X-Forwarded-For' unit test), request.ip authoritative under trust proxy=1 (set in index.ts; ST7 integration tests reject a spoofed leftmost XFF entry), and the 'unknown' fallback is defensive, not a bypass. No action required.
- apps/api/src/common/throttle/throttle.service.ts:81-96 - INFO — FAIL-CLOSED determination: checkRequest() does NOT wrap store.hit() in a try/catch, so a store error (e.g. a future Redis timeout) propagates out of checkRequest -> out of the guard's canActivate -> NestJS returns 500 = request DENIED. This is fail-CLOSED, the correct posture for an anti-abuse control. Off-by-one is correct: the store returns count including the current hit and the service blocks on count > maxHits, so exactly maxHits requests succeed and the (maxHits+1)th is the first 429 (matches AC 'over-limit -> 429'; covered by at-limit/over-limit tests).
  Directly answers R2's fail-open vs fail-closed question: as written the control is fail-closed on store error and has no off-by-one defect. The only caveat is that this guarantee is not pinned by a test (see the test-sufficiency CONCERN above).
