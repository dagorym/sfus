## Acceptance Criteria

AC1: Published standalone pages render at top-level paths (e.g., /about); the root catch-all is evaluated last and never shadows existing or reserved routes; reserved slugs are rejected on create/edit.

AC2: Only published pages are public; draft/unpublished revisions remain protected.

AC3: Every edit creates a durable revision capturing summary, change note, editor user, and featured media; restore creates a new revision (audit trail preserved).

AC4: Page bodies are sanitized/normalized server-side on create/update/restore; featured media uploads via ImageUpload and renders publicly; current-body resolution no longer scans the full revision list per fetch.

AC5: No block-builder or wiki behavior is introduced.

## Key Behaviors To Test

### AC1 — Top-level routing and reserved-slug enforcement
- Reserved slugs must be rejected by `PagesService.assertSlugValid()` (create and update) with BadRequestException
- Test each slug in the reserved list: admin, api, app, blog, login, register, onboarding, profile, settings, health
- Hyphenated slugs that start with a reserved word but are not exact matches (e.g., `admin-notes`) must be accepted
- The new `app/[slug]/page.tsx` component must import `getPublishedPage`, not admin helpers
- It must render a not-found state for reserved slugs without hitting the API
- It must use `MarkdownRenderer` for body content
- It must render a featured image when `featuredMediaId` is present

### AC2 — Published-only public visibility
- `findPublishedBySlug` must only return pages with `status = "published"`
- The public route must render a not-found state when page is null

### AC3 — Durable revision metadata
- `PagesService.create()` must store summary, changeNote, featuredMediaId in the revision
- `PagesService.update()` must set editorUserId on new revisions and capture summary/changeNote/featuredMediaId
- `PagesService.restoreRevision()` must set editorUserId, auto-set changeNote to reference the source revision, and carry forward summary/featuredMediaId from the source revision
- `RevisionDetail` in the API response must include editorUserId, summary, changeNote, featuredMediaId

### AC4 — Sanitization and efficient body resolution
- `PagesService.create()` must reject bodies containing unsafe patterns (script tags, event handlers, javascript:)
- `PagesService.update()` must reject unsafe bodies
- `pages-client.ts` PageDetail interface must include `summary` and `featuredMediaId`
- `pages-client.ts` RevisionDetail interface must include `editorUserId`, `summary`, `changeNote`, `featuredMediaId`
- The controller's `resolveCurrentBody` must not call `findRevisions` (full list scan); it must use `findRevisionById`
- The edit and new admin pages must import `ImageUpload` from the shared components

### AC5 — Scope discipline
- No block-builder, wiki, document-tree constructs in any changed file
