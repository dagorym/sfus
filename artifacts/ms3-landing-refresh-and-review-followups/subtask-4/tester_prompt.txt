## Implementation Summary

A publication-leakage bug was fixed in `GET /api/blog/:postId/comments`. Previously, the UUID-fallback path in `BlogController.listComments()` checked only `status === "published"` when resolving a post by id, which allowed future-scheduled posts (published but `publishedAt` in the future) addressed by UUID to return HTTP 200 with an empty comments array instead of 404.

**Changes made:**

1. **`apps/api/src/blog/blog.service.ts`** — Added `findPublishedById(id: string): Promise<BlogPostEntity | null>` with the same `status: "published", publishedAt: LessThanOrEqual(now)` predicate as `findPublishedBySlug`.

2. **`apps/api/src/blog/blog.controller.ts`** — Updated `listComments()` UUID-fallback from the incomplete `findById(...).then((p) => p?.status === "published" ? p : null)` to `findPublishedById(postId)`, which enforces the full two-part visibility gate.

## Acceptance Criteria

- AC1: GET `/api/blog/:postId/comments` for a future-scheduled post (status=published, publishedAt in the future) addressed by UUID returns 404.
- AC2: Comment listing for genuinely public posts (status=published, publishedAt <= now) is unchanged, both by slug and by UUID.
- AC3: Draft/unpublished posts continue to return 404 on this route.

## Implementation Context

- `BlogService.findPublishedById()` is the new method. It takes a post `id` (UUID) and applies the same `LessThanOrEqual(now)` predicate as `findPublishedBySlug`. Returns `null` for draft, unpublished, or future-dated posts.
- `BlogController.listComments()` (line ~244): first tries slug lookup via `findPublishedBySlug(postId)`, then falls back to `findPublishedById(postId)`. Both now enforce the full visibility predicate. If neither resolves a publicly-visible post, throws `NotFoundException`.
- The fix is a two-line controller change + one new service method. No new routes, no schema changes, no migration required.
