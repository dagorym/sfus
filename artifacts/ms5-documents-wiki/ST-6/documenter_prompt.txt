## What changed (implementation facts — update docs to match these)

### 1. acquireLock 200 response shape (BLOCKING-2 fix)

The controller method `acquireLock` (POST /api/docs/:id/lock) now returns the
flat `DocsLockResultShape` directly:

```json
{
  "pageId": "<page-id>",
  "lock": {
    "isLocked": true,
    "lockedByUserId": "<user-id>",
    "lockedAt": "<ISO timestamp>",
    "lockExpiresAt": "<ISO timestamp>"
  }
}
```

There is NO wrapper object (the previous code had an extra `result:` envelope
at this endpoint). The lock state fields are directly under the `lock` key.

### 2. acquireLock 409 error response shape (BLOCKING-1 fix)

When the page is already locked by another user, the API returns 409 with:

```json
{
  "error": {
    "statusCode": 409,
    "code": "CONFLICT",
    "message": "This page is currently locked by another user.",
    "details": {
      "lockedByUserId": "<holder-user-id>",
      "lockExpiresAt": "<ISO timestamp>"
    }
  },
  "request": { ... }
}
```

The lock holder metadata is under `error.details`, NOT at a top-level `lock`
key. `error.details.lockedByUserId` and `error.details.lockExpiresAt` are the
canonical field paths.

### 3. softDeletePage is now transactional (WARNING-1 fix)

The `DELETE /api/docs/:id` endpoint now runs the entire delete operation
(page lookup, foreign lock check, child count check, status update) inside a
single database transaction. The in-transaction lock check ensures that an
active foreign lock on the page will cause the soft-delete to throw 409
(ConflictException) even under concurrent writes.

### 4. JsonExceptionFilter now forwards `error.details` (BLOCKING-1 fix)

The global `JsonExceptionFilter` has been updated to extract the optional
`details` field from an HttpException response body and emit it at
`error.details` in the JSON error envelope. This is now part of the API
error shape contract:

- If the exception body is an object with a `details` key, that value is
  included as `error.details` in the response.
- If the exception body has no `details` key (or is a plain string), there
  is no `error.details` field in the response (backward compatible).
